# Prompt Review

Use when the user supplies existing website prompts and asks for analysis, synthesis, or a better replacement.

## Review Steps

1. Identify repeated structure, stack, visual language, and motion patterns.
2. Extract what works: specificity, strong section ideas, useful stack details, good constraints.
3. Identify what feels generic or repeated.
4. Write a new master prompt that preserves the quality level but changes the concept grammar.

## What to Compare

- First viewport.
- Navigation model.
- Section order.
- Visual system.
- Motion model.
- Stack and dependency roles.
- Asset direction.
- Implementation constraints.
- Anti-pattern constraints.

## Review Rubric

Rate only as much as needed to write a better replacement:

- Specificity: concrete product, audience, action, visual system, and content.
- Buildability: stack, file structure, component expectations, states, and QA checks.
- Uniqueness: site grammar, first viewport, IA, motion model, and media direction differ from common templates.
- Stack clarity: every library has a role and no unnecessary dependencies are included.
- Visual clarity: typography, palette, layout, surfaces, and media are precise enough to implement.
- Responsive clarity: desktop, tablet, mobile, touch, overflow, and density behavior are covered.
- Anti-placeholder strength: prompt forbids filler content, broken assets, generic sections, and vague adjectives.

## Replacement Prompt Rules

- Do not summarize all old prompts unless the user asks.
- Do not produce a minor rewrite.
- Use a fresh business/theme when allowed.
- Include exact stack, fonts, palette, media, interactions, file structure, and responsive rules.
- Make the new prompt copy-ready.
- Keep useful ideas from the old prompt, but change weak structure, vague language, and missing constraints.
