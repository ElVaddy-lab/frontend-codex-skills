# Anti-Generic Design Rules

Use this reference when a brief risks becoming a generic landing page or decorative trend stack.

## Reject Generic Structure

Do not default to:

```text
navbar -> hero -> feature cards -> testimonials -> CTA -> footer
```

Replace it with a structure based on the product scenario:

- SaaS/dashboard: workspace shell, status summary, filters, main work table/chart, detail panel, action rail.
- Tool UI: work surface first, toolbar/inspector, preview/result area, history/settings/export.
- Ecommerce: product or catalog inspection, variant/filter controls, comparison/trust, cart confidence.
- Portfolio/editorial: archive/index, case-study entry, media viewer, narrative proof, contact surface.
- Service: proof-first opening, process path, service fit, booking/contact, risk reduction.

## First Viewport Rule

The first viewport must show at least one of:

- actual product/interface state;
- a product/object/work item that can be inspected;
- the main tool surface or primary useful action;
- operational status, data, or workflow context;
- a strong editorial lead with evidence preview.

Avoid decorative hero screens where the product/category is only implied by vague copy.

## Banned Decorative Defaults

Avoid unless the task gives a specific reason:

- floating gradient orbs;
- decorative blobs or bokeh;
- generic glassmorphism panels;
- purple/blue one-note gradients;
- stock-like people imagery that does not reveal product, work, place, or state;
- oversized marketing hero type inside compact app/tool surfaces.

## Specificity Tests

Before finalizing, the brief should answer:

- What makes this IA different from a template?
- What does the first viewport prove?
- What visual asset or interface state carries the page?
- Which controls, states, or comparisons make the design useful?
- What should not be built because it would make the result generic?
