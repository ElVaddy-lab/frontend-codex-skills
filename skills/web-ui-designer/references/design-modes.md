# Design Modes

Use this reference to choose the brief mode before writing the design brief.

## Brief Modes

| Mode | Use when | Output emphasis |
| --- | --- | --- |
| `concept brief` | The user has an early product idea, niche, audience, or goal but no settled interface direction | Assumptions, non-goals, IA, first viewport concept, visual lane |
| `production brief` | The user wants a complete Web/UI design direction for a real site, app, dashboard, store, or tool | Full 10-section brief with components, states, responsive, accessibility, and handoff |
| `redesign brief` | Existing site/mockup/code has been analyzed or described and the user wants a new direction | Preserve useful evidence, define what changes, name risks and unknowns |
| `handoff-ready brief` | The next step is planning, implementation, or generator prompt creation | Concrete design decisions, component/state expectations, constraints, and handoff status |

## Product Scenario Rules

- **Landing/content**: prove category fit quickly, make the next action clear, and use semantic content, evidence, and media rhythm.
- **SaaS/dashboard**: prioritize repeated use, dense but readable information, predictable navigation, visible state, filters, tables, and empty/loading/error states.
- **Ecommerce/catalog**: prioritize product inspection, comparison, variants, price/trust signals, delivery/stock clarity, cart confidence, and mobile buying flow.
- **Portfolio/editorial**: prioritize identity, inspectable work, typography, media rhythm, archive/detail movement, and memorable navigation.
- **Tool UI/web app**: put the actual work surface, editor, preview, canvas, form, or control model in the first experience.
- **Service business**: prioritize trust, process clarity, proof, service fit, booking/contact path, and risk reduction.

## Assumptions vs Blocking Questions

Make assumptions when:

- the missing detail can be reasonably inferred from product type or audience;
- the user asked for momentum rather than research;
- the assumption can be clearly marked and corrected later.

Ask or hand off first when:

- the request depends on an existing interface that has not been inspected;
- brand/legal/compliance constraints are essential;
- the primary user action is unknown and multiple actions would produce different IA;
- the user asks for implementation, stack, dependencies, or file structure.

## Mode Output Rule

Always state:

- Brief mode;
- Product scenario;
- Handoff status.

Use `Not material for this brief` for sections that do not apply to a small concept brief.
